# Care-Bridge AI - 시스템 아키텍처 & 파이프라인 설계

## 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | Care-Bridge AI |
| 설명 | 멀티모달 LLM 기반 병동 간호사 인수인계 자동 요약 및 환자 위험도 스크리닝 시스템 |
| 스택 비용 | 전체 무료 ($0/월) |

---

## 1. 기술 스택

| 레이어 | 서비스 | 용도 | 무료 범위 |
|--------|--------|------|-----------|
| 프론트엔드 | Vercel (Next.js) | 웹 UI + 음성 녹음 | Hobby 무료 |
| 백엔드 | Render (FastAPI) | API + 파이프라인 오케스트레이션 | 750시간/월 |
| LLM | Groq (LLaMA-3.1-8B) | SBAR 요약 생성 | 분당 30요청 |
| STT | Groq (Whisper large-v3) | 음성→텍스트 | 25MB/파일 |
| DB | Supabase PostgreSQL | 환자/기록 데이터 | 500MB |
| 인증 | Supabase Auth | 로그인/역할 관리 | 50,000 MAU |
| 파일저장 | Supabase Storage | 음성 파일 보관 | 1GB |
| 슬립 방지 | Vercel Cron + cron-job.org | Render/Supabase 깨우기 | 무료 |

## 4. 핵심 모듈 상세

### 4-1. STT 모듈

| 항목 | 내용 |
|------|------|
| 서비스 | Groq Whisper large-v3 API |
| 엔드포인트 | `POST https://api.groq.com/openai/v1/audio/transcriptions` |
| 지원 포맷 | mp3, wav, webm (최대 25MB) |
| 언어 | 한국어 (`language: "ko"`) |
| 출력 | 변환된 텍스트 문자열 |

```
호출 흐름:
프론트(녹음) → Supabase Storage 업로드 → 백엔드가 파일 다운로드 → Groq API 전송 → 텍스트 반환
```

### 4-2. LLM 요약 모듈 (SBAR Generator)

| 항목 | 내용 |
|------|------|
| 서비스 | Groq LLaMA-3.1-8B (또는 70B) |
| 엔드포인트 | `POST https://api.groq.com/openai/v1/chat/completions` |
| 출력 포맷 | SBAR JSON |

```
SBAR 출력 구조:
┌────────────────────────────────────────────────────────┐
│ S (Situation)      : 현재 상황 요약                     │
│   - 환자명, 병실, 주호소, 현재 상태                    │
│                                                         │
│ B (Background)     : 환자 배경/병력                     │
│   - 입원 사유, 과거력, 알레르기, 현 투약               │
│                                                         │
│ A (Assessment)     : 간호 평가                          │
│   - 활력징후 변화, 이상 소견, 검사 결과 요약           │
│                                                         │
│ R (Recommendation) : 권장 조치                          │
│   - 주의 관찰 항목, 예정된 처치, 의사 보고 필요 여부   │
└────────────────────────────────────────────────────────┘
```

### 4-3. 위험도 스코어링 엔진

| 항목 | 내용 |
|------|------|
| 방식 | 규칙 기반 (Modified Early Warning Score) |
| GPU 필요 | 없음 (순수 Python 로직) |
| 입력 | Vital Sign + Lab 수치 |
| 출력 | 점수 + 등급 (Critical/High/Medium/Low) |

```
스코어링 규칙:
┌───────────────────────────────────────────┐
│  항목              │ 조건        │ 점수   │
├───────────────────────────────────────────┤
│  SpO2             │ < 92%       │ +3     │
│  수축기 혈압      │ < 90mmHg    │ +3     │
│  심박수           │ > 130 bpm   │ +3     │
│  체온             │ > 38.5°C    │ +2     │
│  호흡수           │ > 25회/분   │ +2     │
│  의식수준 변화    │ 변화 있음   │ +3     │
├───────────────────────────────────────────┤
│  총점 ≥ 7점      → Critical (즉시 조치)  │
│  총점 5~6점      → High (1시간 내 확인)  │
│  총점 3~4점      → Medium (모니터링)     │
│  총점 0~2점      → Low (정상 관찰)       │
└───────────────────────────────────────────┘
```

### 4-4. 우선순위 큐 (Next Action Generator)

| 항목 | 내용 |
|------|------|
| 방식 | 가중치 기반 우선순위 알고리즘 |
| 입력 | SBAR 요약 + 위험도 + 시간 제약 |
| 출력 | 정렬된 업무 리스트 |

```
우선순위 계산식:
Priority = (위험도 × 0.4) + (시간긴급도 × 0.3) + (의존성 × 0.2) + (복잡도 × 0.1)

출력 카테고리:
  URGENT  → 즉시 수행 (투약 시간 임박, Critical 환자 대응)
  HIGH    → 1시간 내 수행
  NORMAL  → 근무 중 수행
  LOW     → 여유 시 수행
```

## 12. 무료 서비스 제한사항 & 대응

| 서비스 | 제한 | 대응 |
|--------|------|------|
| Render | 15분 미활동 시 슬립 | Vercel Cron 10분 핑 |
| Supabase | 1주 미활동 시 일시정지 | 같은 Cron에서 DB 쿼리 |
| Groq | 분당 30요청 | 동시 요청 큐잉 + 리트라이 |
| Groq | 음성 파일 25MB | 프론트에서 압축/분할 |
| Supabase Storage | 1GB | 처리 완료 후 오래된 파일 삭제 정책 |
| Vercel | Serverless 10초 | 무거운 처리는 Render에서 |
